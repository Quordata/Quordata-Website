Quordata Process Overview

This post is meant to give a high-level overview of the querying and analysis process. For terminology and concepts that are unfamiliar, it is highly recommended the reader performs external research to appreciate the full body of work.

User Query

The Quordata process begins with a simple query input. This input can be anything, but for the purposes of this post we will operate under the assumption that the queries are based on companies. When a query is submitted, the first step is looking through our database of cached results. If we have recently processed this query, the results are simply retrieved from the cache. These results come in a proprietary structure that is then formatted to a clean and readable UI, which will be discussed later.

If, however, the query is not cached then the real work starts. Quordata will continue by looking in our NoSQL database and doing a reverse document lookup to find all entries related to the query. This data comes from many different sources. Quordata collects information from Twitter, Reddit, LinkedIn, Wall Street Journal, New York Times, Washington Post, Yahoo Finance, and various other news outlets. These are then consolidated into CSV, JSON, and Parquet files for later processing. Quordata is constantly collecting data, and sometimes a query may have little to no relevant data stored. In these cases, Quordata notes frequently requested queries and queries with little information to ensure priority collection later. This will be explained in further detail in another post.

Spam Model

With our data consolidated Quordata passes it through to the first model, the spam model. Quordata combines textual data with metadata to evaluate the likelihood of data being spam. Metadata can include followers, retweets, and likes for tweets; upvotes, downvotes, replies, and awards for Reddit posts; and comments and page hits for news articles. Quordata uses this metadata with the text data to look for patterns to predict when something might be considered spam. Spam is not the same as irrelevant data, but rather spam is data that is used for the express purpose of advertising, promotion, is sent by a bot, or otherwise repeating a meaningless message.


Unless otherwise stated, all the Quordata machine learning code is written in Python. The spam model utilizes the Tensorflow and HuggingFace libraries. The spam model uses a pretrained BERT model as a base, with transfer learning protocols to train the model to our specific use case. We manually labeled thousands of data points following the above rules to feed into our model. Quordata begins with the pretrained model and then takes in our manually labeled data to generate a model that can accurately predict whether a data point is spam. The model is a binary classification model. That is, our spam model will predict whether an input aligns with one of two classification tags: not spam (ham) or spam. The model actually outputs two values, the probability of a point being each label. If the model is 60% confident that an input is spam, it returns the values 0.6 and (1 - 0.6) = 0.4. That means that the model is 40% confident that the input is ham. Quordata uses what is known as a softmax calculation to choose the appropriate label for an input given the two confidence levels.

Sentiment Model

Afterwards, the same input data is passed into the sentiment model. This model focuses purely on the textual data, no metadata is used for labeling. Unlike the spam model, the sentiment model is known as a multiclassification model, with three possible labels. These are positive, negative, and neutral. The sentiment model also receives several thousand manually labeled data points for training. First, the model is based on a pretrained SiEBERT model from the HuggingFace library. Quordata again uses transfer learning to fine tune the model. For both the spam and sentiment models, our hyperparameters have been tweaked and continue to be analyzed and tweaked as necessary. As before, the model outputs three values for each label. We use a softmax probability distribution to choose the appropriate label given the confidences of each label.

Biterm Topic Model

The last model utilized is the Biterm Topic Model (BTM). This model is meant to generate closely related topics to a given document set. In this case, our document set is all the data collected regarding the initial query. Our document set is passed into the BTM and the output is a collection of phrases that are related to each other given the scope of the document set. Multiple collections of phrases are gathered. For example, a query such as “Apple” may have one collection containing phrases like “ios, macbook, laptop” and another “apple music, itunes, ipod”. Each of these collections of phrases are used to predict prevailing unique topics about the base query. In this case, “Apple” might receive topics like “macbook” and “music”. Quordata returns the top five most relevant topics this way. These topics can then be further utilized for more relevant data querying or further user research.

Analysis Results

Now that our data has been labeled with spam and sentiment models, Quordata is ready to create the analysis results. Various metrics such as average sentiment score, confidence levels, and source counts can be generated with simple arithmetic on the results. The closely related topics are retrieved from the BTM. Quordata can also relate model scores to stock prices and trading volume, assuming the query is business-related. Stock data is received from Yahoo Finance, with backup libraries if necessary. Quordata can make trading suggestions based on past historical performance in relation to the sentiment. Using recent trending information, Quordata can also make predictions based on what may be driving current price and sentiment action. These timeframes are based on the frequency of the data collected and the amount of data available for the given query.

User Friendly Dashboards

Finally, all this data is collected and passed through several different tools to generate user-friendly presentations. Chart.js, HTML, CSS, Python’s Plotly library, and jquery scripts are the main drivers behind the graphical user interface shown on a dashboard. When the GUI is ready, the server sends the results back to the user that initialized the query for their viewing pleasure.
